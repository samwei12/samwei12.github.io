### 第一阶段，找出一本书在谈什么

#### 用最简短的句子说出整本书在谈些什么

* 这是一本关于如何编写具有高可读性代码的书

#### 按照顺序与关系，列出全书的重要部分。将全书的纲要拟出来之后，再将各个部分的纲要也一一列出

* 作者先概括的介绍了可读代码的重要性:
  1. 代码应当易于理解: **可读性基本定理**
* 之后介绍第一部分,表面层次的改进:
  2. 把信息装到名字里
     1. 选择专业的词:避免使用"空洞"的词
     2. 避免泛泛的名字
     3. 用具体的名字代替抽象的名字
     4. 使用前缀或后缀来给名字附带更多信息
     5. 决定名字的长度:
        1. 在小的作用域里可以使用短的名字
        2. 避免项目特有的缩写
     6. 利用名字的格式来表达含义
  3. 不会误解的名字:避免二义性
     1. min/max
     2. first/last
     3. begin/end
     4. 布尔值避免使用反义
  4. 审美
     1. 使用一致的布局,让读者很快习惯这种风格
     2. 让相似的代码看上去相似
     3. **把相关的代码行分组,形成代码块** (**很实用且重要**)
  5. **该写什么样的注释**
     1. **了解什么不需要注释**
        1. 不要为那些从代码本身就能快速推断的事实写注释
        2. 不要为了注释而注释
        3. 不要给不好的名字加注释---而应该把名字改好
     2. **用代码记录你的思想**
        1. 加入"导演评论": 对于为什么代码写成这样而不是那样的内在理由
        2. 对代码中的瑕疵写注释: 使用像TODO：或者XXX：这样的标记
        3. 给常量加注释: 常量背后的故事，为什么是这个值
     3. 站在读者的角度,去想象他们需要知道什么
        1. 意料之中的提问: 预料到代码中哪些部分会让读者说：“啊？”并且给它们加上注释
        2. 公布可能的陷阱: 为普通读者意料之外的行为加上注释
        3. "全局观"注释: 在文件/类的级别上使用“全局观”注释来解释所有的部分是如何一起工作的
        4. 总结性注释: 做什么、怎么做、为什么: 用注释来总结代码块，使读者不致迷失在细节中
        5. 最后的思考---克服"作者心理阻滞"
  6. 写出言简意赅的注释
     1. 让注释保持紧凑
     2. 避免使用不明确的代词
     3. 润色粗糙的句子
     4. 精确地描述函数的行为
     5. **用输入/输出例子来说明特别的情况**
     6. 声明代码的意图
     7. "具名函数参数"的注释
     8. 采用信息含量高的词
* 第二部分,简化循环和逻辑:
  7. 把控制流变得易读: 把条件、循环以及其他对控制流的改变做的越“自然”越好。运用一种方式使读者不用停下来重读你的代码。
     1. 条件语句中参数的顺序: **被判断的变量放左边,偏向常量的值放右边,不需要"尤达表示法"了,也就是倒装句**
     2. if/else语句块的顺序
        1. 首先处理正逻辑而不是负逻辑
        2. 先处理掉简单的情况
        3. 先处理有趣的或者是可疑的情况
     3. ?:条件表达式("三目运算符"):默认情况下都用 if/else,三目运算符只在最简单的情况下使用
     4. 避免 do/while 循环
     5. 从函数中提前返回(注意资源释放,使用 RAII 可以避免该问题)
     6. 臭名昭著的 goto
     7. 最小化嵌套: 
        1. 当你对代码做改动时,从全新的角度审视它,把它作为一个整体来看待
        2. 通过提早返回来减少嵌套
        3. 减少循环内的嵌套
  8. 拆分超长的表达式: 把你的超长表达式拆分成更容易理解的小块(DRY)
     1. 用作解释的变量
     2. 使用德摩根定理
  9. 变量与可读性
      1. 减少变量
          1. 没有价值的临时变量
          2. 减少中间结果
          3. 减少控制流变量
      2. 缩小变量的作用域
          1. **C++中 if 语句的作用域**
          2. 在 JavaScript 中创建"私有"变量
          3. JavaScript 全局作用域
          4. 在 Python 和 JavaScript 中没有嵌套的作用域
          5. 把定义向下移(也就是说用到的时候再声明)
      3. 只写一次的变量更好(操作一个变量的地方越多,越难确定它的当前值,**尽量使用常量**)
* 第三部分,重新组织代码:
  10. 抽取不相关的子问题: 把一般代码和项目专有的代码分开(参考《Refactoring:Improving the Design of Existing Code》中"抽取方法")
  11. 一次只做一件事: 可以尝试把所有的任务都列出来,其中一些编程单独的函数或类,其他的成为一个函数中的逻辑"段落",难点在于如何准确的描述所有的这些小事情 ![CleanShot 2019-04-20 at 22.10.51.png](https://i.loli.net/2019/04/20/5cbb28b872d7a.png)
      1. 任务可以很小
      2. 从对象中抽取值
  12. 把想法变成代码
      1. 清楚地描述逻辑, 用自然语言描述解决方案
      2. 了解函数库是有帮助的
  13. 少写代码: 最好读的代码就是没有代码
      1. 保持小代码库
         1. 创建越多越好的"工具"代码来减少重复代码
         2. 减少无用代码或没有用的功能
         3. 让你的项目保持分开的子项目状态
         4. 小心代码的"重量"
      2. 重新考虑需求,解决版本最简单的问题,只要能完成工作就行
      3. 熟悉你周边的库: **每隔一段时间,花15分钟来阅读标准库中所有函数/模块/类的名字,包括 STL、Java API、Python 内置的模块以及其他内容(比如 Qt API)**
* 第四部分,精选话题:
  14. 测试与可读性
      1. 使测试易于阅读和维护
         1. 是这个测试可读
         2. 创建最小的测试声明
         3. 实现定制的"微语言"
         4. 让错误消息具有可读性
      2. 选择好的测试输入: 应当选择一组最简单的输入,它能完整地使用被测代码
         1. 简化输入值:又简单又能完成工作的测试值更好
         2. 一个功能的多个测试
      3. 给测试函数起名字
      4. 要易于改动和增加新的测试
  15. 设计并改进"分钟/小时计数器"

#### 找出作者要问的问题，或作者想解决的问题

* 想要解决的问题:如何编写高质量的代码?

### 第二阶段，诠释一本书的内容

* 步骤：
  1. 诠释作者使用的关键字，与作者达成共识
  2. 从最重要的句子中抓出作者的重要主旨
  3. 找出作者的论述，重新架构这些论述的前因后果，以明白作者的主张
  4. 确定作者已经解决了哪些问题，还有哪些是未解决的。在未解决的问题中，确定哪些是作者认为自己无法解决的问题

#### 确定作者已经解决了哪些问题，还有哪些是未解决的？

* 是什么让代码变得更好?
  * 别人理解它的时间最小化
* 总是越小越好吗?
  * 尽管减少代码行数是一个好目标，但把理解代码所需的时间最小化是一个更好的目标
* 理解代码所需的时间是否与其他目标有冲突?
  * 不会
  * 可读性基本定理总是有限域其他条例或者原则

### 第三阶段，像是沟通知识一样地评论一本书

> 这本书说的有道理吗？是全部有道理，还是部分有道理？

* 规则：
  1. 除非你已经完成大纲架构，也能诠释整本书了，否则不要轻易批评
  2. 不要争强好胜，非辩到底不可
  3. 在说出评论之前，你要能证明自己区别得出真正的知识与个人观点的不同
  4. 证明作者的知识不足
  5. 证明作者的知识错误
  6. 证明作者不合逻辑
  7. 证明作者的分析与理由是不完整的
