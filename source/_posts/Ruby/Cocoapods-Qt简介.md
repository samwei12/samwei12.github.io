## 背景

柔性供应链团队肩负着集团供应链向上游深水区（此次超链本期宏观）挺进的重任，为此[织女项目](https://www.atatech.org/articles/141687#)应运而生。

怀揣着这个梦想，我们开始了织女的漫漫长征之路，在这个过程中当然遇到了各种坑，就不再赘述了，这里主要介绍下整个开发过程中CI（Continuous Integration）的包管理实践之路。

### 为什么要搞持续集成？

其实在开发初期我们确实是没有CI的，代码都在一个仓库，编译构建都在本地，这种方式适合项目初期的快速搭建。慢慢的，随着代码量增大，组内同学人数增加，问题逐步暴露：
  - 性能：编译时间越来越长，全量编译甚至达到了**30分钟以上**
  - 安全：代码权限无法隔离
  - 稳定性：本地编译，稳定性当然也没法保证了

基于这几个问题，明确了我们需要一个类似于摩天轮的构建体系，那么如何做呢？

### 包管理

构建体系中有个最重要的核心流程和前提，就是 **包管理**，做过移动端开发的同学应该比较了解，就是类似iOS的 **[Cocoapods](https://cocoapods.org/)** 或者 Java的 **[Maven](https://maven.apache.org/)**的东西，它的主要作用就是对不同模块（或者称为包）进行版本管理，解决版本冲突，这样大部分模块使用预编译好的二进制即可，解决了安全和性能问题。

那么选用哪种包管理方案呢？

### 技术选型

首先，我们来分析下我们的具体需求和场景，我们的诉求如下：

- 跨平台且支持Qt：织女使用的是基于C++的Qt框架进行开发的（需要同时提供Mac和Windows端）
- 支持私有仓库：安全性考虑

经过我们的一些简单调研，集团内部的一些PC端软件，大多是**双端两套代码**，很少有像我们这样，使用Qt进行跨平台开发构建的，现有的一些体系，也不太能满足我们的需求。所以我们进行了一些适用于Qt的外部开源方案调研。

下面是一些包管理工具调研报告：

| 名称            | [qpm](https://github.com/Cutehacks/qpm) | [qt-pods](https://github.com/qt-pods/qt-pods) | [Cocoapods](https://github.com/CocoaPods/CocoaPods) | [hunter（CMake）](https://github.com/ruslo/hunter) |
| --------------- | --------------------------------------- | --------------------------------------------- | --------------------------------------------------- | -------------------------------------------------- |
| 活跃状态        | 基本无人维护                            | 无人维护                                      | 活跃                                                | 活跃                                               |
| 私有仓库        | 不支持                                  | 支持                                          | 支持                                                | 不支持                                             |
| **版本冲突**    | 不支持                                  | 不支持                                        | 支持                                                | 不支持                                             |
| **编译入侵性 ** | 无                                      | 无                                            | 无                                                  | 强依赖CMake且入侵性强                              |
| **扩展性**      | 差                                      | 差                                            | 较好                                                | 差                                                 |

身为iOS端开发的我第一时间就想到了Cocoapods，它有这么几个优势：
  - Ruby编写，编译型语言，方便快速迭代，Ruby本身支持跨平台
  - 支持**插件扩展**
  - 设计理念较为轻量，**对编译构建基本无入侵**
  - 摩天轮的iOS与MacOS构建已经支持Cocoapods，接入成本较低
  - 有相关基础，掌握起来成本低

其他类似的工具，比如 **CMake**， 也可以通过 [hunter](https://github.com/ruslo/hunter) 等来进行支持包管理，但是对整体的构建流程会有一定的改造和入侵，不光需要一定的接入和学习成本（要求组内同学均需要掌握CMake语法），而且每次发布版本需要提MR给GitHub，扩展性不好，不便于快速迭代，对后续与摩天轮等集团体系打通也会不够平滑。相反，Cocoapods 基本对构建流程无入侵，单纯只是一个包管理，学习使用成本较低，大家只需要简单了解几个常用字段（例如模块名、作者、版本、依赖路径）即可，不需要对Ruby有深入了解，如果有机会进行集团内迁移或者共建，也十分平滑。

所以我们就愉快的决定在Cocoapods基础上进行扩展来满足业务需求了。

## Cocoapods Qt实践之旅

### 设计理念

技术选型完毕之后，就开始实际的开发工作了，再次明确下我们所需要的扩展内容，由于Cocoapods本身已经支持私有仓库，那么我们需要扩展的内容就是：
![Cocoapods不支持Qt](https://img.alicdn.com/tfs/TB1zMbNaEGF3KVjSZFoXXbmpFXa-329-231.png)

主要是：
  1. 支持Qt平台
  2. 支持Windows
  3. 其他一些自定义功能（例如集成相关）

在这个过程中，我们有几种方式实现自定义需求：
  1. 通过脚本方式内置 Pod 相关环境
  2. 利用 Cocoapods 提供的插件机制进行扩展

下面我们分别分析下这两种方式的优缺点：

#### 内置Pod环境

git仓库中直接内置Cocoapods相关代码，在原有代码中进行修改以满足业务需求，并且通过添加脚本可以达到一键安装和更新的目的，大致示意图如下：
![](https://img.alicdn.com/tfs/TB1RvDTaBKw3KVjSZFOXXarDVXa-451-270.png)

目前iOS中tbenv也是采用类似的方式进行的，这么做的好处包括：
  - 快速开发，代码改动十分便捷
  - 可以内嵌一整套环境，例如Windows下自带zip命令等
  - 版本控制，大家都使用相同版本开发，规避很多环境问题

劣势也比较明显：
  - 脱离Cocoapods体系，兼容性不好
  - 侵入性较强，维护成本高，无法享受最新版本新特性
  - 版本升级较为麻烦，只能通过git拉取方式进行

#### Pod插件

另一种方式则相对优雅一些，利用Ruby自身的加载机制实现，大致示意图如下：
![](https://img.alicdn.com/tfs/TB1X06PaB1D3KVjSZFyXXbuFpXa-462-230.png)

这种方式的好处有：
  - 无侵入，版本兼容性较好，可以同时支持多个版本
  - 可以随时享用到Cocoapods最新版本feature，只需要修改对应的插件hook内容即可
  - 升级方式可以使用原生gems，较为友好
  - 尽可能保持原有理念
  - **未来可以考虑开放共建，便于其他人参与**

缺点：
  - 短期工作量相对直接代码修改要大一些

经过评估调研之后，我们考虑到后续扩展性和复用性，决定采用插件方式进行实现。

### 扩展点

#### 插件机制

在开始动工之前，我们先来了解整套插件机制是如何运转的。这部分内容详见 [Cocoapods插件机制浅析](https://www.atatech.org/articles/141790)

#### 跨平台

真正开始做了之后发现并没有想象中的简单，首先，`Cocoapods` 的改造遇到的第一个问题是如何进行跨平台支持，因为 Ruby本身虽然是跨平台的，但是 `Cocoapods` 开发过程中，并未考虑其他平台，Windows平台下安装执行就是一个大问题。

例如，未声明`utf-8`，`run`中耦合Xcode相关等等，
```ruby
def self.run(argv)
    help! 'You cannot run CocoaPods as root.' if Process.uid == 0

    verify_minimum_git_version!
    verify_xcode_license_approved!

    super(argv)
ensure
    UI.print_warnings
end
```
这会导致 `pod` 命令压根无法执行成功，自然也就无法加载相关插件，我们后续的各种扩展自然就无从谈起了。

这里目前我们采用专门新增一个称为 `wpatch`的gem用于完成这个工作。具体的思路也比较简单，检测用户目录下所有 pod 版本，自动将阻塞 `run` 方法运行的代码进行注释或者替换，确保 `pod` 命令完整执行，这其中还做了很多其他工作，例如Windows 命令行中的语法高亮等，详情可以自行拉取对应 gem 进行查看。

#### Qt扩展

插件机制可以正常启动之后，剩下的事情相对来说就比较好办了。

这里，我们的有一个理念 **尽可能保持Pod原汁原味**，所以大部分Pod中的设计思路我们选择了保留，例如`target`、`workspace`、`sandbox`等等。

##### 基础设施

首先，我们需要一套类似于[Xcodeproj](https://github.com/CocoaPods/Xcodeproj)的工具类，这里我们提供了两套相关基础类，分别用于便捷操控Qt的`pro`和`pri`文件。

##### 资源包

C++或者Qt都没有自己明确的包格式，也就是没有明确定义资源文件跟二进制文件应该如何组合，这里我们自行约定了一套格式，大致格式如下：

```bash
module.qtlib/
    include/
    lib/
        macx/
            x86_64/
                debug/
                release/
        win/
            ...
    rsrc/
```

二进制这里按照 `平台/架构/编译模式` 进行存储。每个产物包同时包含两个平台的二进制，这样做的好处就是可以**共用同一个坐标依赖**，避免版本割裂；缺点也比较明显，任意平台改动都需要升级tag。

##### 核心

其次，利用Qt工程的Subdir来达到Pod中原本的XcodeWorkspace功能，这样可以做到对原本的`Pod`最小改动。一个通用的工程结构大概是下图这样：
![](https://img.alicdn.com/tfs/TB15y9FaHus3KVjSZKbXXXqkFXa-434-406.png)
- 每个项目`App.pro`对应生成一个`App.ws.pro`文件，也就是Workspace文件
- 源码依赖，例如`ModuleA`，会自动生成一个pro文件，产物会被`Pods.pri`依赖
- 二进制依赖，例如`ModuleB`，会自动生成一个pri文件，被`Pods.pri`依赖

至此，整个工程基本已经可以正常运作了。

- ![](https://img.alicdn.com/tfs/TB1IkSxaSSD3KVjSZFKXXb10VXa-1728-1080.gif)

#### 项目扩展

下面简单介绍下项目过程中定制化的一些扩展。

##### 持续集成

团队中有相关移动端背景的同学比较少，因此第一个遇到的问题就是整体的打包构建流程略繁琐，一个发布流程包括：

1. 开发feature
2. 升级podspec的版本
3. 发布pod
4. 升级git的tag

由于我们当时没有类似摩天轮这样的完整构建体系支撑（经过沟通，摩天轮目前也没有支持Windows平台的规划），所以上述步骤还是给大家日常开发流程带来了一些麻烦的，为了解决这个问题，我们自行部署了一套Jenkins环境，并且开发了从代码构建到静态扫描、单元测试、最终发布的CI体系。

为了让大家平时开发时候可以快速接入这套构建体系，我们对Pod进行了自定义扩展，可以直接在执行pod命令时候完成整个发布流程。效果图如下：

![](https://img.alicdn.com/tfs/TB1UT.XaUCF3KVjSZJnXXbnHFXa-1728-1080.gif)

##### 软链接改造

这样开心的过了一段时间之后，随着项目代码量越来越庞大，二进制产物体积也直线上升，整体的 `pod update` 速度慢慢成为了一个大问题。尤其是在Windows平台下，由于机械硬盘和文件系统本身的差异，速度更是慢的出奇，只 `pod update` 一项基本保持在 **11分钟** 左右，整体打包时间在 15分钟左右，占比达到 **70%以上**。

经过排查之后发现，原始版本的Cocoapods中，存在多次拷贝（省略了部分细节）:

![](https://img.alicdn.com/tfs/TB1_5j8aL1H3KVjSZFBXXbSMXXa-174-428.png)

这里在从tmp目录到Cache目录和从Cache目录到Pods目录分别有一次全量拷贝，在IO较慢的机器上十分影响体验。

这里的设计初衷可能是为了不同工程之间相互不依赖，且Cache被清空之后不影响已经构建完的工程使用；但是带来的负面影响就是磁盘占用翻了N倍，每个工程里面都有一份，且`pod update`速度过慢。

经过仔细评估，我们的认为速度和磁盘空间对我们来说收益大于Cache清空之后重新update一次的成本，因此我们将这部分逻辑改造如下（省略了部分细节）：

![](https://img.alicdn.com/tfs/TB1sZZXaQWE3KVjSZSyXXXocXXa-174-428.png)

提速前后对比：

![](https://img.alicdn.com/tfs/TB1ilVga21H3KVjSZFBXXbSMXXa-1670-1032.jpg)

可以看到，在Windows下，提升幅度巨大，全量更新时耗时大约只有原始版本的 **18%**！

### 整体示意图

经过这几轮的扩展改造之后，Pod这边整体插件结构如下：

![](https://img.alicdn.com/tfs/TB1Y2kjaRKw3KVjSZFOXXarDVXa-474-283.png)

- Cocoapods-wpatch： Windows平台扩展，目前包含一个可执行文件，检测并对已安装 Cocoapods 进行patch，确保Pod命令可以正常执行。
- Cocoapods-Qt： Qt 相关扩展，包括 pro、pri 文件生成修改，Workspace 生成等。
- Cocoapods-aliscm：供应链团队在跨平台开发过程中，自行定制扩展的部分。
